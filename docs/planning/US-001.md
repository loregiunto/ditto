## User Story

**Epic:** EP-001 — Listing Management
**Priorità:** HIGH | **Story Points:** 5

**Story**
Come Giulia (host), voglio creare un listing del mio bagno con foto, descrizione, indirizzo, prezzo orario e tipo di host, così da rendere disponibile il mio spazio sulla piattaforma.

**Criteri di Accettazione**
- [ ] Form con campi: titolo, descrizione, tipo host (Privato / Attività Commerciale), indirizzo (geocodificato via Mapbox), prezzo orario
- [ ] Upload fino a 10 foto via Supabase Storage; le foto sono associate al listing
- [ ] Il listing viene salvato in DB con status `draft`
- [ ] L'indirizzo esatto è salvato nel DB ma esposto solo agli utenti con prenotazione confermata
- [ ] Validazione: tutti i campi obbligatori presenti prima di poter salvare

---

## Soluzione Tecnica

L'approccio si appoggia al boilerplate esistente (Next.js App Router, Supabase, Prisma, shadcn/ui) e introduce Mapbox come unica nuova dipendenza esterna; modelliamo il listing come entità autonoma con due campi indirizzo distinti — uno completo riservato e uno pubblico di sola zona — così da preparare strutturalmente la privacy richiesta dai criteri senza anticipare logica di esposizione che spetta a US-005/US-008. Il form è un client component basato su `react-hook-form` + `zod`, le foto sono caricate direttamente dal browser su Supabase Storage in un path drafts dell'utente (gli URL sono persistiti al submit), e la geocodifica avviene server-side al momento della creazione tramite un helper Mapbox dedicato.

- **Modello dati:** nuovo `Listing` (id, hostId → User, title, description, hostType enum `PRIVATE`/`BUSINESS`, addressFull, addressDisplay, latitude, longitude, hourlyPriceCents Int, status enum default `DRAFT`, timestamps) + `ListingPhoto` (id, listingId, url, order) — relazione 1:N con `onDelete: Cascade`
- **Storage foto:** bucket Supabase `listing-photos` con policy "auth owner only" sul path `{userId}/...`; upload diretto client → Supabase via client autenticato; il path drafts (`{userId}/drafts/{uuid}/...`) viene referenziato as-is dal record DB al submit, senza spostamenti
- **Geocoding:** helper server-only `src/lib/mapbox.ts` che chiama Mapbox Geocoding API con `MAPBOX_ACCESS_TOKEN`, deriva `addressDisplay` ("Quartiere, Città") dal campo `context` della response e ritorna `{addressFull, addressDisplay, lat, lng}`; autocomplete UI usa `NEXT_PUBLIC_MAPBOX_TOKEN` restricted
- **API:** `POST /api/listings` autenticato via `getCurrentUser()`, body validato con `zod`, geocodifica server-side, `prisma.listing.create` con nested `photos.createMany`; ritorna `{ id }` del listing creato
- **Frontend:** pagina `/host/listings/new` (server shell) con client component `<ListingForm>` strutturato in sezioni — tipo host (radio), info base (titolo, descrizione, prezzo), indirizzo (input con autocomplete Mapbox), foto (drag & drop con cap 10) — coerente con i mockup esistenti in `docs/Mockup/HomeRest/host-onboarding.jsx`
- **shadcn/ui:** aggiungere i componenti mancanti `form` e `radio-group`
- **Middleware:** estendere il matcher esistente per proteggere anche `/host/*` oltre a `/dashboard`
- **Env vars nuove:** `MAPBOX_ACCESS_TOKEN` (server), `NEXT_PUBLIC_MAPBOX_TOKEN` (client autocomplete)

## Strategia di Test

Test pyramid a tre livelli con coverage interleaved sui task: unit per la logica pura, integration sull'API con DB reale (Supabase test schema o Prisma test container), e un singolo e2e Playwright sull'happy path di creazione listing con upload simulato.

- **Unit** — schema `zod` di input listing: campi obbligatori, prezzo > 0, tipo host valido, lunghezze min/max title/description, max 10 foto
- **Unit** — helper Mapbox: parsing del `context` response, fallback quando manca `place`/`neighborhood`, gestione di indirizzo non geocodificabile
- **Integration** — `POST /api/listings`: happy path crea record con status `DRAFT` e foto associate; rifiuto 401 senza sessione; rifiuto 400 su campi mancanti, prezzo invalido, >10 foto, indirizzo non geocodificabile
- **Integration** — verifica che `addressFull` e `addressDisplay` siano persistiti separati e che una select pubblica del listing non ritorni `addressFull`
- **E2E** — Playwright: signin → naviga a `/host/listings/new` → compila form → upload 2 foto fixture → submit → verifica redirect e listing visibile come bozza nella dashboard host (mock dell'autocomplete Mapbox per evitare network esterno)

## Note di implementazione (post-review)

- **TASK-07 — divergenza intenzionale dal piano:** non sono stati installati `shadcn/form` né `react-hook-form`. Il resto del codebase (`src/app/auth/signin/page.tsx`) usa `<form>` nativo + `useState` + `zod` per la validazione; introdurre RHF per un solo form avrebbe spezzato la coerenza. La validazione zod è comunque condivisa tra form e API in `src/lib/validation/listing.ts`. I radio button usano `<input type="radio">` nativi stilizzati con Tailwind.
- **Post-review hardening:** la API ora accetta `photos[].path` (non URL) e verifica che ogni path inizi con `{user.supabaseId}/`; in caso contrario risponde 403. La public URL viene costruita server-side. L'estensione del file caricato è derivata dal MIME whitelistato, non dal `file.name`.
- **Privacy strutturale verificata:** test integration `tests/integration/listings-api.test.ts` verifica che `addressFull` e `addressDisplay` vengano persistiti separati e che la public URL sia derivata dal path.
- **Debiti tecnici aperti** (da affrontare in storie successive):
  - cleanup automatico dei file `drafts/` orfani in Supabase Storage
  - valutare bucket privato + signed URLs per le bozze (oggi pubblico)
  - committare la fixture `tests/e2e/fixtures/sample.jpg` quando si avvieranno gli e2e
