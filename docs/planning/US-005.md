# US-005: Mappa interattiva con bagni disponibili — Piano di Implementazione

**Generato da:** Archetipo Planning Team

## User Story

**Epic:** EP-002 — Discovery & Mappa
**Priorità:** HIGH | **Story Points:** 5

**Story**
Come Marco (utente), voglio vedere i bagni disponibili su una mappa interattiva con la mia posizione, così da trovare rapidamente opzioni nelle vicinanze in situazioni di urgenza.

**Criteri di Accettazione**
- [ ] Mappa Mapbox GL JS caricata sulla home page come componente principale
- [ ] Solo i listing con status `active` e slot disponibili nelle prossime ore sono mostrati
- [ ] I listing sono clusterizzati per prossimità; cliccando un cluster la mappa fa zoom
- [ ] Cliccando un pin singolo appare una preview card: nome, foto principale, prezzo/ora, valutazione media, badge tipo host
- [ ] La geolocalizzazione del browser è richiesta al caricamento; se concessa, la mappa centra sulla posizione utente
- [ ] L'indirizzo esatto non è visibile sulla mappa (solo zona/quartiere)

---

## Soluzione Tecnica

L'approccio espone un endpoint server `GET /api/listings/map` che ritorna **solo i campi pubblici** dei listing `ACTIVE` con almeno uno slot prenotabile nelle prossime 6 ore — la home page diventa una shell server che monta un componente client `<MapView>` con Mapbox GL JS in dynamic import (no SSR), sfruttando il **clustering nativo di Mapbox** via sorgente GeoJSON con `cluster: true`. La privacy è garantita strutturalmente: la query Prisma `select` esclude `addressFull` e ritorna solo `addressDisplay`. Il filtro temporale riusa `weeklyRulesToSlots` introdotto in US-003, mantenendo centralizzata la regola "slot disponibili" e già pronto a intersecare con `Booking` quando arriverà US-009/US-010. Coerente con i pattern del repo: API route con auth/zod, client component minimale stilizzato shadcn/Tailwind, no librerie aggiuntive oltre `mapbox-gl`.

- **API `GET /api/listings/map`:** pubblica (no auth richiesta), filtra `status = ACTIVE`, carica le `availabilityRules`, per ogni listing applica `weeklyRulesToSlots(rules, now, now + 6h, 30, [])` e scarta chi non ha alcuno slot. Risposta: `{ listings: [{ id, title, addressDisplay, latitude, longitude, hourlyPriceCents, hostType, photoUrl }] }`. La query usa `select` esplicito — `addressFull` non viene mai letto né serializzato.
- **Costanti condivise (`src/lib/listings/discovery.ts`):** `AVAILABILITY_WINDOW_HOURS = 6` e `FALLBACK_MAP_CENTER = { lng: 12.4964, lat: 41.9028, zoom: 12 }` (Roma) — riusabili da US-006/US-007.
- **Home page (`src/app/page.tsx`):** trasformata in shell server full-bleed che monta `<MapView>` (client) e una topbar minimale con logo HomeRest e i controlli auth esistenti (Sign In / Dashboard / Sign Out).
- **Componente `<MapView>` (client):** dynamic import di `mapbox-gl` con `ssr: false`. In `useEffect`: richiede `navigator.geolocation`, su success centra sull'utente, su denied/error/timeout usa `FALLBACK_MAP_CENTER` mostrando un banner contestuale dismissible. Inizializza la mappa, fetch a `/api/listings/map`, registra una sorgente GeoJSON con `cluster: true, clusterRadius: 50, clusterMaxZoom: 14`.
- **Layer Mapbox:** un layer `circle` per i cluster (con `text` count), un layer `symbol` per i pin singoli (icona SVG inline custom HomeRest). Click su cluster → `getClusterExpansionZoom` + `flyTo`. Click su pin singolo → mostra `<ListingPreviewCard>` come popup ancorato al pin.
- **`<ListingPreviewCard>`:** card shadcn con foto principale, titolo, `addressDisplay`, prezzo `€{hourlyPriceCents/100}/h`, badge `hostType` (🏠 Privato / 🏪 Commerciale), placeholder rating (`Nuovo` finché non esistono review — campo previsto per US-015). CTA "Vedi dettagli" disabilitata con tooltip "Disponibile a breve" (US-008 non ancora implementata).
- **Privacy strutturale:** test integration verifica esplicitamente che la response API non contenga mai la stringa di `addressFull` né il campo. La query Prisma `select` impedisce il leak anche per regressione futura.
- **Empty state:** se la response è `{ listings: [] }` un banner overlay "Nessun bagno disponibile nella tua zona nelle prossime ore" rimane visibile sopra la mappa.
- **Bundle hygiene:** `mapbox-gl` caricato solo lato client e solo sulla home (dynamic import). I CSS Mapbox sono importati dentro il client component, non in `globals.css`.

## Strategia di Test

Test pyramid coerente con US-001/US-003: unit per la pura logica di proiezione (riuso del helper esistente, solo nuovi casi al margine), integration sull'API con DB reale, e un singolo e2e che verifica payload + DOM dei pin senza dipendere da WebGL.

- **Unit** — `discovery.ts` costanti e helper di shape della response (mapping `Listing` → DTO pubblico): conferma che `addressFull` non sia presente nelle chiavi e che `photoUrl` sia derivata dalla prima foto in `order` ascendente
- **Unit** — `weeklyRulesToSlots` con finestra di 6 ore: listing con regola attiva ora → almeno uno slot ritornato; listing con regole solo in giorni futuri oltre 6h → nessuno slot, listing scartato
- **Integration** — `GET /api/listings/map`: ritorna solo listing `ACTIVE` (esclude `DRAFT` e `INACTIVE`); esclude listing `ACTIVE` senza availability nelle prossime 6h; la response non contiene `addressFull` per nessun listing (controllo su tutto il body serializzato); 200 anche senza sessione (endpoint pubblico)
- **Integration** — verifica che la prima foto (per `order`) sia esposta come `photoUrl` e che listing senza foto restituiscano `photoUrl: null`
- **E2E** — Playwright: apri `/` → mock di `/api/listings/map` con 2 listing fixture → la mappa monta (verifica presenza container `[data-testid="map-view"]`) → click su un marker (selettore stabile via attributi DOM custom) → preview card visibile con nome, prezzo, badge tipo host. Geolocalizzazione browser mockata per evitare prompt permessi.
- **E2E** — flusso "geolocalizzazione negata": override `navigator.geolocation` → banner di fallback visibile, mappa centrata sul fallback Roma
